# v1.154.409 (2026-07-22)

perf(browse): precompute the event-date aggregates so startdate/enddate sort by index. Last two slow browse sorts, ~22-23s a page on atom.theahg.co.za. Different shape from the others: they do not order by a column at all, they order by MIN(event.start_date) / MAX(event.end_date), which needs a LEFT JOIN to event plus a GROUP BY across all twelve selected columns - 10.3s for the result query alone. The sidecar now stores those aggregates precomputed as real DATE columns, start_date_sort and end_date_sort, each with a full index on (culture, <col>, object_id), so the sorts join nothing and group nothing. Only 52,249 of 454,393 records have a dated event; the rest stay NULL and sort exactly where MIN()/MAX() put them, so null placement is unchanged. The join and GROUP BY are now skipped entirely when the sidecar answers - they were only ever there to produce the sort key, and the GROUP BY would additionally have collapsed the FIELD() ordering the id-first page depends on. COLUMNS grew a per-column DDL type so ensureColumns() adds DATE rather than VARCHAR for these two, and alignCollation() skips them naturally since neither side has a collation - date ordering is not collation-dependent. Verified identical over 12 combinations - both sorts, both directions, pages 1, 2 and 50 - against the old GROUP BY query, and every sort/direction pair smoke-tested
