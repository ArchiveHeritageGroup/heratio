# ADR-0002: AI inference provenance discipline baseline

- **Status:** Accepted
- **Date:** 2026-05-04
- **Issue:** [ArchiveHeritageGroup/heratio#61](https://github.com/ArchiveHeritageGroup/heratio/issues/61)
- **Supersedes:** none
- **Superseded by:** none

## Context

GitHub issue #61 sets the bar: every AI inference must be **one query away** from being defensibly traced to which model, which decision (input → output), which confidence, which human override, which standard. A read of the codebase showed the current state is greenfield against that bar:

- `ahg-ric/SparqlQueryService` is read-only; nothing currently writes RDF (let alone RDF-Star) into Fuseki.
- `ahg-provenance` is for archival chain-of-custody (donor, owner, acquisition), unrelated to AI inference.
- `ahg-audit-trail` covers user actions (file access, authentication); has no AI-inference table.
- `ahg-workflow` is record-state machinery; has no AI-correction hook.
- `ahg-ai-services` (NER, HTR, Translation, LLM, Donut) writes outputs directly into MySQL tables (e.g. `object_term_relation`) with no model/version/confidence/timestamp metadata.

This is an opportunity, not a setback: there is no legacy provenance pattern to migrate, so we can design the inference contract from first principles.

This ADR locks the seven architectural decisions that gate the rest of the work. They are load-bearing - reversing any of them later costs significant rework.

## Decision

### 1. Provenance lives in MySQL **and** in Fuseki (mirror)

- MySQL `ahg_ai_inference` is the operational store: fast filtering, status counts, drift dashboards, the admin UI.
- Fuseki RDF-Star is the canonical defensible semantic record: SPARQL queries, FOIA traces, semantic-web interop.
- Every write goes through `InferenceService::record()`, which writes the SQL row first, then enqueues a Fuseki RDF-Star insert. If the Fuseki write fails, the SQL row carries `fuseki_graph_uri = NULL` and the row is flagged for replay.

**Why dual-store and not pure-Fuseki:** ops needs to count "all inferences in last 24h with confidence < 0.6" without round-tripping SPARQL; reviewers need fast paginated lists; dashboards need joins to user/IO tables that are SQL-native. Fuseki alone forces all of that into SPARQL gymnastics that are slow at scale.

**Why not pure-MySQL:** the FOIA-defensibility argument requires semantic chains expressible in standard ontologies (PROV-O, RiC-O). RDF-Star is the right substrate for inference-as-meta-assertion. SQL alone cannot express "this triple was generated by this activity attributed to this agent at this time" without ad-hoc schema.

### 2. Override semantics: hybrid - RDF-Star for inference, PROV-O reified Activity for override

- The inference itself is recorded as an RDF-Star meta-assertion on the generated triple:
  ```turtle
  <<:io123 :subject "Mining">> prov:wasGeneratedBy :inference456 ;
                                prov:generatedAtTime "2026-05-04T10:00:00Z"^^xsd:dateTime ;
                                ex:confidence "0.87"^^xsd:decimal ;
                                ex:model "spaCy en_core_web_sm 3.8.0" ;
                                ex:standard "ICIP-subject-indexing" .
  ```
- Overrides are reified as full PROV-O Activities (Shape B from the plan), pointing at the original inference:
  ```turtle
  :override789 a prov:Activity ;
      prov:used :inference456 ;
      prov:wasAssociatedWith :user42 ;
      prov:atTime "2026-05-04T11:00:00Z"^^xsd:dateTime ;
      ex:originalValue "Mining" ;
      ex:newValue "Hard rock mining" ;
      ex:reason "More specific term per IO content" .
  ```

**Why hybrid:** RDF-Star is compact and natural for "this triple has metadata." But overrides have richer human-agent semantics (reviewer, reason, before/after) that read more clearly when reified as a `prov:Activity`. Auditors and FOIA officers do not speak RDF-Star; they recognise PROV-O activities.

**Original triples are NEVER overwritten.** When a reviewer corrects an AI inference, the original assertion stays in place and the override is added alongside. The "current effective value" is computed from the chain at query time.

### 3. Inference subject is per-field, not per-record

- Each inference targets a specific field on a specific entity: `(target_entity_type, target_entity_id, target_field)`.
- One IO can have many inferences across different fields, generated at different times by different services.
- The override layer also operates per-field, so a reviewer who corrects only `subject` does not invalidate the AI's `physical_characteristics` inference.

**Why:** matches how the AI services actually work. NER targets specific cells (subject, person access points). HTR targets `physical_characteristics` or full-text-content. Translation targets `*_i18n.<field>`. Per-record granularity would lose information.

### 4. Standard attribution is inference-site, not record-level

- Each inference declares the standard it is targeting via the `standard` column / `ex:standard` predicate.
- Examples: NER → `ICIP-subject-indexing` or `Spectrum-5.1-subject`. HTR → `ISAD(G)-physical_characteristics`. Translation → `Heratio-i18n-NLLB-200` (or whatever model is in play). LLM summary → `RiC-O-scope_and_content`.

**Why inference-site:** the standard is a property of the production decision, not the record. The same record can have one field generated against ISAD(G) and another against Spectrum 5.1; only the inference knows which.

### 5. Confidence threshold is per-service, configured in `ahg_settings`

- New settings keys (in setting_group `ai_provenance`):
  - `ai_provenance.ner.confidence_review_threshold` (default 0.7)
  - `ai_provenance.htr.confidence_review_threshold` (default 0.85, since CER is more stringent)
  - `ai_provenance.translation.confidence_review_threshold` (default 0.6)
  - `ai_provenance.llm.confidence_review_threshold` (default 0.5)
  - `ai_provenance.donut.confidence_review_threshold` (default 0.7)
- An inference at or above the threshold auto-applies. Below, it is queued for human review via `ahg-workflow` (Phase 3).

**Why per-service:** confidence distributions differ wildly across model families. A spaCy NER score of 0.7 is high; an NLLB-200 BLEU-equivalent of 0.7 is mid-quality. One global threshold is meaningless.

### 6. Forward-only - no back-fill of historical inferences

- Once Phase 1 ships, every new AI write logs full provenance.
- Existing rows in `object_term_relation`, `information_object_i18n`, `museum_metadata_i18n`, etc. that were AI-generated before this work landed remain unprovenanced. They are treated as "manually authored" because we cannot reconstruct which model wrote them or with what confidence.
- A future cleanup task (out of scope for issue #61) could batch-tag legacy rows with a sentinel inference of model=`legacy-ai-pre-provenance` if the academic argument requires it.

**Why:** retroactive reconstruction is forensic guesswork. Better to start the clock than to falsify a paper trail.

### 7. Fuseki update endpoint goes in `config/heratio.php` + `ahg_settings`

- Read endpoint stays as `config('heratio.fuseki_endpoint')`, default `http://localhost:3030/heratio` (existing).
- New `config('heratio.fuseki_update_endpoint')`, default `http://localhost:3030/heratio/update`.
- Optional `ahg_settings` keys for credentials when Fuseki is behind auth: `fuseki.update_username`, `fuseki.update_password` (plain-text, treated as a service credential per existing `ftp_password` pattern - acknowledged technical debt).
- RDF-Star capability is REQUIRED; Apache Jena Fuseki ≥ 4.x supports it natively. Phase 1 includes a startup check that fails loud if the configured endpoint does not accept RDF-Star insert.

## Consequences

### Positive

- **Defensibility-by-construction.** Every AI write goes through one service that enforces the contract; "we forgot to log it" cannot happen because the AI services no longer have direct write access to target tables (they go through `InferenceService::recordAndApply()` or its variants).
- **Single trace endpoint.** Phase 4's `GET /api/v1/provenance/{entity}/{id}/trace` is a thin wrapper over the dual store - SQL for the chain summary, SPARQL for the canonical RDF-Star triples.
- **Forward-only is honest.** No academic risk of fabricated retroactive provenance.
- **Override pattern is auditor-legible.** PROV-O reification reads naturally to FOIA/legal officers without RDF-Star fluency.

### Negative

- **Two writes per inference.** SQL + Fuseki round-trip on every AI call. Mitigation: queue Fuseki via `php-redis` or DB queue in Phase 1.5 if synchronous adds latency; for now, blocking writes are acceptable.
- **Override flow has to thread through every entity edit form.** Phase 3 will touch ~6 entity controllers' `update()` methods. Bounded but real.
- **Legacy AI rows are unprovenanced.** Some users may want their pre-2026-05 AI-suggested terms back-tagged. We will revisit if that becomes a real ask.
- **Fuseki becomes a runtime dependency.** Existing read-only Fuseki use means it is already deployed in production; this just makes it write-path-critical. Outage of Fuseki must not break record edits - SQL row writes succeed, Fuseki write is queued for replay.

### Neutral

- **New package** `ahg-provenance-ai` rather than extending `ahg-provenance`. Names are confusing but renaming `ahg-provenance` (which has live archival CoC users) is not worth the breakage.
- **No GraphQL exposure in Phase 1.** Trace endpoint will be REST (Laravel route + JSON response). GraphQL hookup can come later via `ahg-graphql` if needed.
