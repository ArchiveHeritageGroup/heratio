# v1.154.411 (2026-07-22)

perf(browse,protocol): skip protocol gates that cannot exclude anything, and memoise the browse counts. With every sort fixed, the two counts were the whole remaining page cost - 3,951ms for the result total and 2,089ms for the digital-object total against ~3ms for everything else. Two changes. (1) TermProtocolGate::excludeRestrictedRecords attached three correlated NOT EXISTS subqueries unconditionally whenever the tables merely EXISTED. On atom.theahg.co.za all three were empty, so they filtered nothing and cost ~1,792ms per count (3,719ms with, 1,927ms without). Each gate now also checks the table holds at least one row that actually restricts - a whereNotExists against a table with no restricting row is always true, so skipping it is exactly equivalent, not an approximation. The ICIP gate checks APPLIED labels rather than the label vocabulary, because the type table ships ~22 Local Contexts labels of which several restrict, so testing it alone would keep the gate on for every install even where no record has ever been labelled. Memoised per REQUEST only, never across requests, so a protocol applied by an editor takes effect on the very next page load; unreadable tables fail CLOSED. TermProtocolGateTest 8/8, including 'guest cannot see record tagged with restricted term', which is what proves the gate still enforces where rows do restrict. (2) Both counts now go through cachedCount(), keyed on the query's own SQL and bindings rather than a hand-listed set of filter fields - there are 30-odd filters and applyFilters() also branches on authentication, so an enumerated key would be one forgotten field away from serving a guest a count computed for an editor. Verified the guest and authenticated keys differ (and their counts do too, 359 vs 441 on dev), and that each filter combination keys separately. TTL is operator-tunable via the new ahg_display_count_cache_ttl setting, default 300s, 0 to disable
