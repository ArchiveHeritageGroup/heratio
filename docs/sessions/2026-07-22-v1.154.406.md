# v1.154.406 (2026-07-22)

feat(access-request): scope a request to everything, a collection, or a single item. /access-request/new offered no way to say WHAT access was wanted - every request was written with the hardcoded default scope_type='single' and nothing ever read it, so reviewers had only the free-text description to go on. The storage was already there and unused: access_request.scope_type, plus an access_request_scope satellite carrying object_id, object_title and include_descendants. This wires the form to it. Three scopes: Everything (no target); A collection (include_descendants=1, so it covers everything catalogued inside); A single item (include_descendants=0). 'Collection' is defined by level of description - Collection/Fonds/Subfonds/Series/Subseries - NOT by being top-level: on atom.theahg.co.za 418,541 records sit directly under the root, which is an unusable dropdown, while only 65 records are collections by level, which is both selectable and the archivally correct meaning. Items are chosen through the existing informationobject/autocomplete endpoint via a dependency-free datalist; the hidden id is what the server trusts and it is cleared whenever the text stops matching a suggestion, so a half-typed title cannot be submitted as though a record had been picked. Server-side validation requires a target for the two scoped forms (required_if) and ignores it for 'all'. The chosen title is denormalised onto the scope row so reviewers still see what was asked for if the record is later renamed or deleted. Scope is now visible where decisions are made: a new Scope column on my-requests and pending, and a Scope row on the request detail page with a link through to the requested record, all rendered from one shared partial. Legacy rows keep scope_type='single' and display as 'Not specified' rather than pretending to a scope nobody chose. Verified end to end as an authenticated user: form renders all three controls and 51 collection options, each scope persists the right scope_type and scope row (descendants 1 for collection, 0 for item, no row for all), a collection request with no target is rejected with no row created, and both listings render all three badges
