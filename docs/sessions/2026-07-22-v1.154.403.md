# v1.154.403 (2026-07-22)

fix(browse): tie-break the alphabetical sort on id so paging is stable. Follow-up to v1.154.402. Verifying the sidecar against the old path on atom.theahg.co.za showed the two returning DIFFERENT SETS at LIMIT 300 ascending - not a projection error: the rows at the boundary had identical sort values ('1006_49.pdf', one of many long runs of identically-named scans in that catalogue), and with no tiebreaker their relative order is whatever the plan happens to emit. That is a real pre-existing bug rather than a cosmetic one: a LIMIT/OFFSET boundary landing inside a tied run can show a record on two consecutive pages or skip it entirely. Adds ->orderBy('io.id') after the title on both the sidecar path and the fallback. Free on the sidecar path - object_id is the third column of idx_iots_culture_title, so it is satisfied by the same index scan. With the tiebreak applied to both, old and new agree exactly: same set and same order, ascending and descending, over 300 rows of real data
