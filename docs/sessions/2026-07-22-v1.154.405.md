# v1.154.405 (2026-07-22)

perf(browse): only join display_object_config into the count when a type filter needs it. Last of the browse passes. With the result query fixed in v1.154.404, the remaining cost was the two counts. applyFilters() touches doc.object_type for the sector/type filter and nothing else, so with no type filter the LEFT JOIN adds no rows and no predicate - it only makes the count walk a second 454k table. Now joined conditionally. Identical answers verified both sides: atom 454,392 with and without, dev 359 with and without; 3,063ms -> 1,894ms on atom. The digital-object count was left alone deliberately - the obvious cheaper shape (counting off digital_object directly) returns 453,526 against the current 453,185, because the join to information_object also drops digital_object rows whose object_id is not a live IO, so it is not an equivalent rewrite. Browse landing on atom is now ~8.5s, down from ~20.7s; sort=identifier is unchanged at ~18s since only the alphabetical path is accelerated
