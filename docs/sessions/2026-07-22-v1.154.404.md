# v1.154.404 (2026-07-22)

perf(browse): pick the alphabetical page off the sidecar index (id-first). v1.154.402/403 added the title-sort sidecar but wiring it as a JOIN delivered NOTHING - measured on atom.theahg.co.za, base column 9,234ms vs sidecar join 9,751ms. The sidecar index is only usable for ordering when the sidecar is the ONLY table in the FROM clause: join information_object to it and MySQL converts the published EXISTS into a semi-join, drives from status (226,944 rows) and falls back to 'Using temporary; Using filesort'. INNER vs LEFT, JOIN_PREFIX, NO_SEMIJOIN and semijoin=off all made no difference (~3.7s each on the reduced shape). Keeping information_object inside a correlated EXISTS instead leaves the outer query single-table, so the plan becomes 'Backward index scan; Using index' and stops once it has a page: browse now resolves its page of ids that way and fetches only those rows, re-imposing order with FIELD(). applyFilters() adds only where-clauses against io.*, so the entire filter set - published gate, #1388 cultural protocols, facets, text search - travels into the subquery unchanged; the display_object_config join goes with it because applyFilters filters on doc.object_type for the sector filter, and without it the filter would throw and the catch would silently revert to the slow path. Verified identical, not merely faster: id-first vs the old joined ORDER BY over pages 1, 2, 17 and 500 in both directions on real data - IDENTICAL every time, 4,500ms -> 1-35ms. Also confirmed the fast path is actually taken rather than falling back, with no filter and with type filters. applyAlphabeticSort() reverts to the base column and now tie-breaks on io.id in the SAME direction as the title, since a mixed DESC/ASC pair cannot be served by a single-direction index. browseAjax still uses the plain path - correct, just not accelerated
