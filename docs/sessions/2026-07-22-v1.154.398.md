# v1.154.398 (2026-07-22)

perf(term): stop term pages sorting a whole taxonomy to draw a sidebar. Follow-up to v1.154.397, which stopped the memory crash but left the page at ~15s on atom.theahg.co.za. Two costs found by profiling. (1) The sibling query orders alphabetically, so MySQL sorted the entire 558,715-row level to return the first 50 - ~4,000ms - and those 50 names are an arbitrary slice of half a million, which is noise rather than navigation. Adds TREE_WIDE (1000): past that width the sibling query is skipped outright and the tree reports the size of the level and points at the List tab. (ORDER BY term.id runs in 2ms but gives a meaningless order, so not drawing it is the right answer.) (2) listTotal counted through two joins it never needed - the term_i18n/slug joins exist to fetch name and slug for the 25 displayed rows, not for a COUNT - which walked 362,095 joined rows for one number: 2,000ms versus 112ms counting off term alone. Neither STRAIGHT_JOIN nor FORCE INDEX helps the remaining listTerms sort (~3,600ms); that needs the List tab to load lazily and is left as a follow-up. Dev unregressed: an 800-wide group is below TREE_WIDE so still draws, HTTP 200, 'and 748 more at this level'
