The problem

ISAD(G) is a tree. Reality is a graph.

A fonds → series → file → item hierarchy can only say "is part of". But a record is also created by an agent, about a place, resulting from an activity, governed by a mandate.

Records in Contexts (RiC) - the ICA's new standard - models all of that as a graph. The question for us: how do we actually see and share it?

OpenRiC2 / 7
Land the "tree vs graph" point hard - every archivist feels the limits of strict hierarchy. RiC-CM + RiC-O are the model; OpenRiC is how you touch it.
What OpenRiC is

A spec + a viewer. Bring your own server.

OpenRiC API

A small, open HTTP spec any archive can implement to serve its holdings as RiC-O graph data.

The Viewer

A browser app that talks to any conformant server - 2D & 3D, no install, no lock-in.

Reference API

A live implementation at ric.theahg.co.za proving the spec against a real archive.

Portable

Fixtures via a service worker mean it even runs offline, against canned data.

OpenRiC3 / 7
The key idea: the viewer is decoupled from any one system. Implement the spec, point the viewer at your endpoint, done. Conformance, not a product silo.

Explore a fonds as a living graph

The relationships are the description - rico: predicates you can follow, not free-text notes.
