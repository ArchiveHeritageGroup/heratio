# v1.154.414 (2026-07-23)

feat(ric,oai): RiC-O as an OAI-PMH metadata prefix (#1425 A2). Adds a standards-proper RiC-O harvesting surface: the main multi-format OAI endpoint (/oai, ahg-oai - the one that already serves oai_dc/ead/ead3/mods/marcxml) now offers metadataPrefix=rico, emitting RiC-O as RDF/XML. Engine gains RicSerializationService::toRdfXml() next to the existing toTurtle(): it maps the serializeRecord()/serializeAgent()/etc JSON-LD mechanically to RDF/XML - scalars -> literals, {'@id'} -> rdf:resource references, {'@value'} -> typed literals, nested maps -> rdf:parseType='Resource' blank nodes - reused by OAI now and available to metadata-export later. OaiPmhController advertises rico in ListMetadataFormats and renders it in GetRecord/ListRecords via renderRico(), all behind formatAvailable() which gates rico on class_exists(RicSerializationService): a minimal/standalone OAI install neither advertises nor accepts a prefix it cannot serve (returns cannotDisseminateFormat), preserving the no-dependencies guarantee. This is a genuine improvement over the pre-existing rico_ld on the RiC-only /api/ric/v1/oai endpoint, which wraps JSON-LD in CDATA (self-described as non-standard) - rico is real RDF/XML on the canonical endpoint. A2 note: the RiC lens for actors/repositories/functions/accessions/rights-holders/digital-objects was ALREADY built (session view-switch + ahg-ric _ric-view-* templates), as are RiC export (ric.crm.export) and import (RdfImportService); this fills the real remaining interop gap. Verified: toRdfXml output is well-formed XML with literals/resource-refs/nested blank nodes; /oai ListMetadataFormats lists rico; GetRecord&metadataPrefix=rico returns a well-formed OAI response carrying rico:title etc; the 6 pre-existing ahg-ric Unit failures (serializeRecord QueryException, degraded test DB) are identical with my change stashed vs applied, so not a regression
