# v1.154.431 (2026-07-24)

v1.154.431: feat(#1425): editable access points on the RiC form (subjects, places, genres, names) + fix a latent persist bug.

The RiC-O form showed Subjects / Places / Genres / Name access points read-only ('none', 'edit via the dedicated widgets' - but there were no widgets), so they could only be edited via the ISAD form. They ARE part of RiC-O (subject -> rico:hasOrHadSubject, place -> rico:hasOrHadSpatialCoverage, genre -> documentary form, name -> agent relation), so the RiC form now lets you add and update all four.

- The read-only 'Access points' badges are replaced with editable autocomplete widgets (the reusable ahg-core::components.autocomplete, the same one MODS/DACS use, which survives the standard-swap), each labelled with its rico property and pre-populated with the record's existing values. Submits subjectAccessPointIds[] / placeAccessPointIds[] / genreAccessPointIds[] / nameAccessPointIds[], which RicManageController::persist already reads.
- Fixed a latent bug this exposed: persist's object_term_relation insert wrote only object_id + term_id, but that table is an AtoM object-subtype needing its own object row + id - so it threw on the first real submission (dead code until now). It now creates the QubitObjectTermRelation object row (matching store()), and on re-save drops the old relations AND their object rows (no orphans, idempotent). fieldsPartial's name query also now selects actor_id so existing names pre-select.

Verified: subject/place/genre/name each save (with backing object rows), re-save is idempotent, existing values pre-populate the widgets, and the serializer emits them - so they flow to every RiC-O surface.
