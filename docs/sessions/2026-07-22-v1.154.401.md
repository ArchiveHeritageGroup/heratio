# v1.154.401 (2026-07-22)

perf(browse): resolve the browse text search once instead of eight times. /glam/browse?query=... took ~100s on atom.theahg.co.za and 502'd at nginx's proxy timeout. The LIKE branch attaches ~9 correlated EXISTS subqueries (i18n title/scope, identifier, DAM, museum, gallery-artist, library item, library creator, authority name) to the browse query, so they are re-evaluated per candidate row of a 454,393-row information_object - and browse runs that same predicate EIGHT times: the result page, the count, and six sidebar facets. Profiling showed 100.3s of the 100.4s was SQL, six facet queries at 13-15s each. Adds resolveTextMatchIds(), which runs the identical branches uncorrelated - one flat scan per table, unioned - memoised per request so all eight consumers share one resolve; applyTextSearchFilter() then filters on whereIn('io.id', ...), an indexed primary key. Verified equivalent, not just faster: old and new id sets compared on atom for apartheid (155), mandela (48), photograph (2,050), 1994 (628) and a no-match term (0) - identical every time, zero missing, zero extra, at 11,300ms -> 1,070ms per execution. Falls back to the old correlated path when the match set exceeds TEXT_MATCH_ID_CAP (20,000, keeping clear of MySQL's 65,535 placeholder limit) or on any exception, so a wide term degrades rather than breaks; an empty set is a real answer and still excludes everything. FULLTEXT path untouched. ALSO restores composer.json/composer.lock to their v1.154.399 state: v1.154.400 swept in the dev-local 'ahg/mva-claims: @dev' glue, whose package directory has zero tracked files and whose vendor entry symlinks outside the repo, so composer install would have failed on prod/atom/sasa
