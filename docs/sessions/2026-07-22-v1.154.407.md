# v1.154.407 (2026-07-22)

perf(browse): index-order the identifier sort too. sort=identifier was the last slow browse path at ~18s a page, and it is the same disease in a worse form: information_object.identifier is varchar(1024) with NO index of any kind, so ORDER BY io.identifier filesorted all 454,393 rows - the title sort at least had a prefix index for MySQL to reject. Extends the existing sidecar rather than adding another: information_object_title_sort gains identifier_sort plus a full-length index on (culture, identifier_sort, object_id), and the id-first page fetch from v1.154.404 is parameterised by column, so identifier/refcode now take the same 'Backward index scan; Using index' route as the title sort. identifier lives on information_object rather than the i18n table, so the same value is stored for every culture of a record - correct, a reference code does not vary by language. Upgrade path handled: instances already carrying the table would never gain a new column, so ensureColumns() adds any missing sort column (and its index) on boot and before each rebuild, guarded by hasColumn so it is a no-op thereafter; available() now checks the specific column exists AND holds a non-null value, so a half-upgraded instance falls back to the base column instead of sorting everything as NULL. alignCollation() generalised to every sidecar column. Verified identical, not merely faster: identifier-sorted pages 1, 2 and 50 in both directions match the old ORDER BY exactly, with 246 of 441 rows carrying a NULL identifier so the null ordering is covered too; collations confirmed matching on both sides. All six sort modes smoke-tested
