# v1.154.388 (2026-07-21)

fix(library): #1417 - drive the FRBR surface off library_item, removing the last library_biblio_* readers and writers. Completes the #1412/#1413/#1414 decision to standardise on library_item; no library_biblio_* reference now remains anywhere in the codebase. FrbrService and FrbrController read through AhgBiblioBf BiblioWorkRepository (Work = library_item work_key cluster, Expression/Manifestation = each library_item, Item = library_copy, Agent = library_item_creator), resolved at call time behind a class_exists guard so ahg-biblio-frbr declares no composer dependency on ahg-biblio-bf - a hard require would be the wrong direction and the surfaces degrade to empty rather than fatal. FRBR is the native consumer of work_key since its own WorkKeyService computes it. Import rewritten to walk the nested Work > realization > Expression > exemplar > Item hierarchy that catalogToXml actually emits, so an Item keeps its Expression and Work; the previous three flat xpath sweeps created every entity unlinked. Works match on title so a re-import does not duplicate, Expressions fill blank publication fields, Items become library_copy rows. Also fixed a pre-existing latent bug: frbrWork cast a Collection with (array), which yields the collection's protected properties rather than its elements, so array_map ran over [items, false] and every work reported two nonsense agents - it was invisible while the phantom tables returned nothing. Added BiblioWorkRepository::createFromImport() as a neutrally-named entry point shared by both importers, with createFromBibframe() delegating to it. Verified on dev: /frbr dashboard reads 16 Works / 19 Expressions (was a permanent 0/0/0), catalogToFrbr returns 6 real agents and 3 expressions correctly clustered, XML export is well-formed, and a full import round trip created title/creator/publisher/date plus a copy and was idempotent on re-run. BIBFRAME unaffected (RDF still well-formed, .ttl 200); ahg-library 198/198
