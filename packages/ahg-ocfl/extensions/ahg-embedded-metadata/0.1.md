# OCFL Community Extension: `ahg-embedded-metadata`

* **Extension Name:** `ahg-embedded-metadata`
* **Version:** 0.1
* **Status:** Draft (vendor extension, not yet submitted to the OCFL Community Extensions registry)
* **Author:** Johan Pieterse / Plain Sailing Information Systems
* **OCFL Spec Compatibility:** OCFL v1.1 (https://ocfl.io/1.1/spec/)
* **License:** AGPL-3.0-or-later
* **Issue Reference:** heratio#753

## 1. Purpose

This extension records the intrinsic, content-embedded metadata (EXIF, IPTC, XMP) extracted from the source digital objects at the time an OCFL version is sealed. It exists so the inventory documents what the object *was* at that point in time, not merely the bytes that make it up.

Per the OCFL v1.1 specification (section 3.7, Object Extensions), the inventory `extensions` object is intentionally open-ended for vendor-defined preservation context. This extension occupies one key under that object: `ahg-embedded-metadata`. It does not modify any spec-mandated inventory field (`id`, `type`, `digestAlgorithm`, `head`, `manifest`, `versions`, `fixity`). Removing the block produces a still-valid OCFL v1.1 inventory.

Use cases:

* **Preservation across format migration.** When a TIFF master is migrated to JPEG2000 the EXIF tags inside the new file may not be byte-identical to the original. The extension carries the originating tags so the migrated version is not severed from the maker, model, capture date, or copyright statement.
* **Fixity audits with semantic context.** An auditor walking inventory.json files can confirm that the IPTC byline / XMP creator / EXIF `Make` recorded against the bytes matches what the catalogue says, without re-reading the file.
* **FOIA / SAR responses.** When a subject access request requires the operator to disclose what intrinsic metadata was held on each version of a record, the inventory itself is the answer (no need to re-extract from a file that may have been format-migrated).

## 2. Block Shape

The extension occupies one key under the inventory's top-level `extensions` object. All sub-blocks except `captured_at` and `extractor_version` are optional.

```json
{
  "extensions": {
    "ahg-embedded-metadata": {
      "exif": {
        "DateTimeOriginal": "2018:07:21 11:43:08",
        "Make":  "NIKON CORPORATION",
        "Model": "NIKON D850"
      },
      "iptc": {
        "byline":           "J. Pieterse",
        "copyright_notice": "(C) Plain Sailing Information Systems"
      },
      "xmp": {
        "creators": ["J. Pieterse", "S. du Toit"],
        "rights":   "All rights reserved"
      },
      "captured_at":       "2026-05-27T08:00:00+02:00",
      "extractor_version": "ahg-metadata-extraction@1.0"
    }
  }
}
```

### Field semantics

| Key                  | Required | Type                          | Notes                                                                                                                                  |
| -------------------- | -------- | ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `exif`               | optional | object (keys alpha-sorted)    | EXIF tags as extracted by the implementation. Tag names are preserved in the case the extractor emits.                                 |
| `iptc`               | optional | object (keys alpha-sorted)    | IPTC IIM / Photo Metadata fields. Multi-value fields (`keywords`, `creators`) MAY be arrays.                                           |
| `xmp`                | optional | object (keys alpha-sorted)    | XMP packet fields, flattened. Multi-value lists MAY be arrays.                                                                         |
| `captured_at`        | required | string (RFC 3339)             | When the extension block was extracted. NOT the capture date of the photo.                                                             |
| `extractor_version`  | required | string                        | Free-text identifier of the extractor (e.g. `ahg-metadata-extraction@1.0`). Lets downstream tools detect upgrade-time field drift.     |

### Determinism requirements

To preserve byte-stability of inventory.json (an OCFL Â§3.5 requirement that two implementations producing the same logical state MUST produce the same bytes):

1. Keys inside each sub-block (`exif`, `iptc`, `xmp`) MUST be alpha-sorted (`SORT_STRING`).
2. Keys at the block root MUST be alpha-sorted.
3. Empty sub-blocks MUST be omitted, not emitted as `{}`.
4. If all three (`exif` + `iptc` + `xmp`) are empty, the entire `ahg-embedded-metadata` key SHOULD be omitted from `extensions`. The reference implementation does this in `EmbeddedMetadataExtension::build()`.

### Privacy gate

GPS-shaped EXIF keys (`GPS*`, `Geolocation*`, `Location*`) MAY be stripped from the block when the operator's privacy regime has flagged the coordinate. In the Heratio reference implementation this is mediated by the `ahg_pii_finding_embedded` table from issue #751 (a pending or escalated `gps_coordinate` finding causes the keys to be omitted). When the privacy table is absent (gate not installed), the extension proceeds without redaction and logs a warning - the "fail open with audit trail" pattern used elsewhere in the Heratio AI / privacy plane.

## 3. Producer Behaviour (informative)

The reference implementation lives in `packages/ahg-ocfl/src/Metadata/` (Heratio):

* `EmbeddedMetadataSource` (interface) - implementations resolve, per OCFL object id, the EXIF/IPTC/XMP rows from sidecar storage.
* `DbEmbeddedMetadataSource` - default implementation. Pulls from `property` rows with `scope='metadata_extraction'` and falls back to `dam_iptc_metadata` for the IPTC block.
* `EmbeddedMetadataExtension::build()` - canonicalises into the block shape and stamps `captured_at` + `extractor_version`.
* `EmbeddedMetadataPiiGate` (interface) + `DbEmbeddedMetadataPiiGate` - the privacy gate described above.
* `StorageRoot::applyEmbeddedMetadataExtension()` - invoked from `StorageRoot::write()` after the new version has been staged. Failures NEVER block the OCFL write (the extension is preservation-enrichment, not fixity-critical).

A backfill artisan command, `ahg:ocfl:backfill-embedded-metadata-extension`, retrofits the extension onto OCFL objects that were ingested before the extension shipped. The backfill emits a metadata-only version bump that inherits the prior `state` + `manifest` and adds the extension block to the new inventory.

## 4. Consumer Behaviour (informative)

A consumer reading an OCFL inventory MAY:

* Cross-reference the `extractor_version` against a known-bad list (e.g. an old extractor version known to mangle certain XMP tags).
* Render the EXIF/IPTC/XMP fields alongside the version's logical content for archivists doing version diffs.
* Surface `captured_at` in the audit trail to show when the contextual metadata was last refreshed (useful when a format migration is months downstream of the original extraction).

Consumers MUST tolerate absence of any field except `captured_at` and `extractor_version`. Consumers MUST tolerate an entirely missing `ahg-embedded-metadata` key (the absence is informative: no extraction was done, or the extractor produced nothing useful).

## 5. Compatibility & Forward Versions

This extension is at draft v0.1. Future v0.x revisions will be backwards-compatible (new optional fields only). A v1.0 will be cut after the extension is submitted to the OCFL Community Extensions registry and a stable URI assigned.
