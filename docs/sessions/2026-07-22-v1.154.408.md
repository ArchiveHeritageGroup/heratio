# v1.154.408 (2026-07-22)

perf(browse): route date/lastUpdated through the id-first path as well. Fixing sort=identifier exposed lastUpdated as the new worst browse sort at ~18s a page. It was already slow before any of this work - it orders by io.id, which IS indexed, but the JOINed query still cannot use that index because the published EXISTS is rewritten as a semi-join that drives from status (226,944 rows) and forces a sort regardless: ~8,500ms for the result query on atom.theahg.co.za. Since ts.object_id IS io.id, the same single-table id-first query answers it off the sidecar's primary key in 2ms, so date/lastUpdated now map to the object_id sort key. available() special-cases object_id: it is the sidecar's own key rather than a projected value, so it needs only that the table exists and is populated, and the tie-break is skipped when object_id is itself the sort key. relevance and startdate/enddate deliberately keep the ordinary path - relevance layers a CASE over the text match and the date sorts aggregate over event with a GROUP BY, so neither reduces to one sidecar column. Verified identical over 454k rows: pages 1, 2 and 50 both directions match the old ORDER BY exactly, 2,000-2,600ms -> 2-7ms. All eight sort modes smoke-tested
